# Flow to Graph Prompt

将任意 pipeline / workflow 转化为可直接投喂给 Gemini 图片生成模型的专业提示词。输出风格固定为 **NotebookLM Style**（干净数字白板感）。

支持两种粒度：
- **汇报粒度**（默认）：适合 PPT 技术汇报，节点为抽象步骤名，不含代码细节
- **设计粒度**：适合技术设计评审，节点标注具体文件路径、函数名、类名，展示代码级调用关系

## Step 0：粒度识别（自动判断，不询问用户）

根据用户的触发语句判断粒度：

| 粒度 | 触发关键词（任一命中即匹配） |
|------|------|
| **设计粒度** | "详细设计图"、"设计图"、"代码级"、"文件级"、"函数级"、"实现细节"、"detailed"、"code-level" |
| **汇报粒度** | 其他所有情况（默认） |

判断后在回复开头标注：`[粒度：汇报]` 或 `[粒度：设计]`。

---

## 工作流（两步，Step 1-2 不可跳过）

### Step 1：与用户共同梳理流程

在生成提示词前，**必须先和用户确认以下内容**。如果用户已提供了控制流文本，则直接从中提取，不足的部分再追问。

#### 汇报粒度 — 确认 5 点（一次性全部问）

1. **主题名称**：这张图的标题是什么？（例："Ascend C QA 语料构建 Pipeline"）
2. **输入层**：有哪些输入源？每个输入源包含哪些关键子项？
3. **核心步骤**：主要处理阶段是什么？哪些步骤需要单独节点，哪些可以压缩合并？
4. **输出层**：最终产物是什么？有哪些类型/分类？
5. **省略项**：有哪些准备步骤、合并步骤或收尾步骤**不需要**出现在图中？

#### 设计粒度 — 确认 7 点（一次性全部问）

前 5 点与汇报粒度相同，**额外增加**：

6. **关键文件与函数**：每个核心步骤对应哪些代码文件？关键入口函数/类是什么？（例：`deploy_ascendc_opp.ts → execute()` 调用 `ascend_tool.py deploy_ascendc_opp`）
7. **调用关系**：步骤之间的代码调用链是什么？哪些是直接函数调用、哪些是通过 tool/CLI/IPC 间接调用？

收到用户回复后，用结构化控制流格式展示理解，请用户确认后再进入 Step 2。

**控制流确认模板（汇报粒度）：**
```
输入层：[源1] + [源2] → 汇合
核心步骤：
  Step A: [名称] — [输入] → [输出]
  Step B: [名称]（子区域：虚线框包裹 B1→B2→B3）
  ...
输出层：[产物1] | [产物2] | ...
省略：[不画的步骤列表]
```

**控制流确认模板（设计粒度）：**
```
输入层：[源1] + [源2] → 汇合
核心步骤：
  Step A: [名称]
    文件：[path/to/file.ts]
    入口：[ClassName.method()] 或 [functionName()]
    调用：[内部调用 path/to/other.py → target_func()]
    输入 → 输出
  Step B: [名称]（子区域：虚线框包裹 B1→B2→B3）
    B1 文件：[...] 入口：[...]
    B2 文件：[...] 入口：[...]
    ...
输出层：[产物1] | [产物2] | ...
省略：[不画的步骤列表]
```

### Step 2：生成 Gemini 图片生成提示词

用户确认控制流后，根据粒度选择对应模板生成提示词。提示词使用中文，直接输出完整文本，用户可以复制粘贴到 Gemini 图片生成模型。

## NotebookLM 风格规范（固定，不可修改）

```
视觉美学：干净、现代、数字白板感，模仿 Google NotebookLM 界面风格
背景：米白色或极淡灰色
节点形状：圆角矩形
线条：柔和，连接箭头清晰
字体：无衬线字体，清晰现代，留白充足

配色方案（按区域）：
  - 输入层：暖灰色
  - 核心处理节点：柔和蓝色系
  - 子区域/并行路径：蓝绿色（新手/主路径）/ 深蓝紫色（专家/次路径）
  - 外部参考/数据库：带小图标的侧挂节点
  - 输出层：柔和橙色
  - 虚线容器（子流程/并行区域）：虚线边框 dashed border

布局方向：从左到右（默认）或从上到下（内容较多时）
所有文字：中文
```

### 设计粒度额外风格规则

在上述基础规范之上，设计粒度**追加**以下规则（不替换基础规范）：

```
节点内部结构（设计粒度专用）：
  - 每个节点分为上下两区：
    - 上区（主标题区）：步骤名称，粗体，正常字号
    - 下区（代码标注区）：文件路径 + 函数名，等宽字体(monospace)，字号缩小一级，颜色略浅
  - 代码标注格式：📄 path/to/file.ts → funcName()
  - 如有多个文件参与同一步骤，下区纵向罗列，每行一个文件

调用关系箭头（设计粒度专用）：
  - 直接函数调用：实线箭头
  - 跨进程/CLI/tool 调用：虚线箭头，旁标注调用方式（如 "subprocess"、"tool call"、"HTTP"）
  - 箭头标签使用等宽字体，标注被调函数签名（如 execute(args)）

模块边界：
  - 不同语言/运行时的代码用不同底色区分：
    - TypeScript 模块：极淡蓝底
    - Python 模块：极淡绿底
    - Shell/CLI：极淡灰底
  - 边界用细实线分隔，顶部小标签标注语言（如 "TS"、"Python"）
```

按以下结构生成提示词，将 `[...]` 替换为用户确认的内容：


## 常见错误

| 错误 | 修正 |
|------|------|
| 跳过 Step 1 直接生成 | 必须先确认控制流，否则节点结构会错 |
| 把合并/收尾步骤画进图中 | 用户明确说"省略"的步骤不出现在提示词里 |
| 输出层节点太简单 | 输出层必须展示类型/分类/维度，不能只写一个框 |
| 忘记虚线框 | 有子流程/并行路径必须用 dashed border 包裹 |
| 配色混乱 | 严格按风格规范的色区分配，不自由发挥 |
| 设计粒度缺少代码标注 | 设计粒度下每个核心节点**必须**有文件路径+函数名的下区标注 |
| 设计粒度箭头不区分 | 直接调用=实线，间接调用=虚线+调用方式标注，不可混用 |
| 设计粒度缺少语言标签 | 不同语言模块必须用底色+顶部标签区分 |
| 汇报粒度写了代码细节 | 汇报粒度不出现任何文件路径、函数名、类名 |
